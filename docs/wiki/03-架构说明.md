# 架构说明

`xiaozhiMCP` 的架构可以理解为一个单入口、多角色、可插拔的 MCP 节点运行时。CLI 只负责配置发现和角色选择，真正的运行逻辑集中在 `app_server.py`，组件加载由 `runtime.py` 负责，远程组件目录由 `catalog_core.py` 负责，跨设备调用由 `cluster.py` 负责。

## 总体结构

```text
xiaozhimcp CLI
  |
  |-- 读取 config.toml
  |-- 自动判断或接收显式 role
  v
app_server.py
  |
  |-- server role
  |     |-- FastMCP("XiaozhiExtensions")
  |     |-- 加载 server 组件
  |     |-- 注册 catalog_* 工具
  |     |-- 注册 logmcp_get_errors
  |     |-- 可选启动 ClusterServer
  |     `-- mcp.run_stdio_async()
  |
  `-- client role
        |-- 加载 client 组件
        |-- export_tools()
        |-- 注册 agent_install_component__{node_id}
        `-- ClusterClient.run_forever()
```

## CLI 层

`cli.py` 暴露 `xiaozhimcp` 命令。它做三件事：

1. 解析子命令：`server`、`client`、`init`。
2. 查找配置文件：当前目录 `config.toml`，然后 `~/.xiaozhimcp/config.toml`。
3. 设置 `sys.argv` 并调用 `app_server.main()`。

角色自动识别逻辑很保守。只有 `[client]` 的三个关键字段都存在且不是占位值时，才判定为 client。否则默认 server。

```python
if client_server_url and client_node_id and client_token:
    return "client"
return "server"
```

## Server 角色

server 角色通过 `FastMCP("XiaozhiExtensions")` 创建 MCP server。启动流程如下：

1. 读取配置。
2. 收集组件。
3. 根据 `supports_role("server")` 过滤组件。
4. 调用每个组件的 `register(mcp)` 注册本地工具。
5. 收集组件 `export_tools()` 返回的工具定义，用于检查与远程工具的名称冲突。
6. 注册 `logmcp_get_errors()`。
7. 如果集群开启，启动 `ClusterServer` 并注册 `cluster_*` 工具。
8. 注册组件目录管理工具 `catalog_*`。
9. 通过 stdio 运行 FastMCP。

server 的核心职责是把本地 MCP 工具集合提供给小智，并作为远程 client 工具的路由中心。

## Client 角色

client 角色不直接连接小智端点，而是连接 server 的 WebSocket 集群入口。启动流程如下：

1. 读取配置。
2. 收集组件。
3. 根据 `supports_role("client")` 过滤组件。
4. 调用组件 `export_tools()` 生成远程工具清单。
5. 追加内置安装工具 `agent_install_component__{node_id}`。
6. 建立 `ClusterClient` 并发送注册包。
7. 监听 server 下发的 `invoke` 请求。
8. 根据工具名分派到组件的 `ainvoke_tool()` 或 `invoke_tool()`。

client 的设计重点是把设备本地能力安全地接入中心 server。它不需要知道小智 endpoint，也不需要暴露 stdio MCP server。

## 组件加载

组件加载由 `runtime.py` 完成。运行时会扫描两个位置：

```text
./components
[components].folder 指定的目录，且该目录不同于 ./components
```

每个扫描项可以是：

- `components/<name>/component.py`
- `components/<name>.py`
- 用户目录下的 `<name>/component.py`
- 用户目录下的 `<name>.py`

实例化顺序：

1. 如果模块有 `build_component(config, full_config)`，优先调用。
2. 否则查找名为 `Component` 的类。
3. 否则查找模块内任意带 `register` 方法的类。

传入组件的 `config` 是同名 TOML section，例如 `[exa]`；`full_config` 是完整配置。

## 组件目录

`catalog_core.py` 从 GitHub 组件仓库读取 `index.json`。正常路径是：

```text
https://raw.githubusercontent.com/{owner}/{repo}/{branch}/index.json
```

如果 `index.json` 不可用，会退回到 GitHub contents API，列出仓库根目录下的文件夹作为组件候选。

安装组件时，目录管理器会下载组件入口和 README：

```text
{component_path}/{entry}
{component_path}/{readme}
```

然后写入目标目录：

```text
components/<component_name>/component.py
components/<component_name>/README.md
components/<component_name>/__init__.py
```

## 集群通信

集群通信基于 WebSocket。server 监听：

```text
ws://<listen_host>:<listen_port>/agent
```

client 首包必须是注册消息：

```json
{
  "type": "register",
  "node_id": "office-windows-pc",
  "client_token": "shared-secret",
  "platform": "windows",
  "tools": [
    {
      "name": "clipboard_get",
      "description": "Read current Windows clipboard text content.",
      "input_schema": {"type": "object", "properties": {}}
    }
  ]
}
```

server 校验 token、`node_id` 和工具名唯一性。注册成功后返回：

```json
{"type": "register_ack", "ok": true}
```

远程调用请求：

```json
{
  "type": "invoke",
  "id": "req-1710000000000",
  "tool": "clipboard_get",
  "arguments": {}
}
```

client 返回：

```json
{
  "type": "invoke_result",
  "id": "req-1710000000000",
  "success": true,
  "result": {"content": "..."}
}
```

## 错误记录

server 会创建 `ErrorStore`，并注册：

```python
logmcp_get_errors(limit=50)
```

集群层会记录常见错误，例如：

| 错误码 | 场景 | 建议 |
| --- | --- | --- |
| `INVALID_REGISTER_PACKET` | client 第一包不是 register。 | 检查 client 协议实现。 |
| `UNAUTHORIZED_CLIENT` | token 不匹配。 | 对齐 server 和 client token。 |
| `MISSING_NODE_ID` | 注册包没有 `node_id`。 | 配置唯一节点名。 |
| `CLIENT_TOOL_CONFLICT` | 多个 client 工具重名。 | 修改工具名前缀。 |
| `CLIENT_TOOL_CONFLICT_WITH_SERVER` | client 工具名与 server 本地工具重名。 | 保留 server 本地工具优先级，修改 client 工具名。 |

## 旧版路由入口

`mcp_router.py` 仍然保留为旧版桥接命令。它的职责是把小智 WebSocket 端点和本地 stdio MCP server 进程连起来：

```text
xiaozhi WebSocket <-> mcp_router.py <-> local stdio server process
```

推荐新部署使用 `xiaozhimcp`。旧入口适合需要兼容 `uv run start` 或已有桥接部署的场景。
