# 开发指南

本页面向希望修改主仓库、调试运行时或开发组件的贡献者。主仓库是一个 Python 3.11 项目，核心代码保持扁平结构，测试位于 `tests/`。

## 本地开发环境

推荐使用 `uv`：

```bash
git clone https://github.com/iiishop/xiaozhiMCP.git
cd xiaozhiMCP
uv sync --extra dev
```

运行测试：

```bash
uv run pytest
```

运行 lint：

```bash
uv run ruff check .
```

启动 server：

```bash
uv run xiaozhimcp server --config config.toml
```

启动 client：

```bash
uv run xiaozhimcp client --config client.toml
```

## 项目结构

```text
xiaozhiMCP/
  README.md
  pyproject.toml
  config.example.toml
  cli.py
  app_server.py
  cluster.py
  runtime.py
  catalog_core.py
  config_loader.py
  config_guide.py
  error_store.py
  mcp_router.py
  tests/
```

模块职责：

| 模块 | 职责 |
| --- | --- |
| `cli.py` | 命令行入口、配置发现、角色选择。 |
| `app_server.py` | server/client 主运行时、MCP 工具注册。 |
| `cluster.py` | WebSocket 集群注册、心跳、远程调用。 |
| `runtime.py` | 组件发现、模块加载、组件实例化。 |
| `catalog_core.py` | 组件目录读取、搜索、安装。 |
| `config_loader.py` | TOML 加载和嵌套配置读取。 |
| `config_guide.py` | 交互式配置生成。 |
| `error_store.py` | 错误持久化和查询。 |
| `mcp_router.py` | 旧版小智 WebSocket 到 stdio MCP 桥接。 |

## 开发原则

- 核心运行时只保留基础设施能力。
- 平台能力、第三方 API、领域功能优先放入组件。
- 工具名必须全局唯一，组件应使用稳定前缀。
- 配置字段应能在 `config.example.toml` 和交互式向导中解释清楚。
- 运行时错误应能通过 `logmcp_get_errors()` 给出可操作结论。
- 网络和文件操作要有明确超时、错误返回或日志。

## 修改 CLI

`cli.py` 的主要入口是 `main()`。修改 CLI 时要注意：

- `init` 子命令直接调用 `config_guide.interactive_setup()`。
- `server` 和 `client` 子命令应显式覆盖自动识别。
- 未指定子命令时，应保持向后兼容的自动识别逻辑。
- `--config` 既支持全局参数，也支持子命令参数。

示例调试：

```bash
uv run xiaozhimcp --config ./config.toml
uv run xiaozhimcp server --config ./server.toml
uv run xiaozhimcp client --config ./client.toml
```

## 修改 server/client 运行时

`app_server.py` 是主流程文件。server 修改重点：

- `run_server(config_path)` 中注册本地组件、错误工具、集群工具和目录工具。
- 新增内置工具时，确认工具名不会与组件工具冲突。
- 如果工具依赖集群，必须只在 `cluster_enabled` 且 `cluster_server` 存在时注册。

client 修改重点：

- `run_client(config_path)` 中收集组件 `export_tools()`。
- client 注册内置安装工具 `agent_install_component__{node_id}`。
- 远程调用优先使用 `ainvoke_tool()`，没有时退回 `invoke_tool()`。

## 修改集群协议

集群协议在 `cluster.py`。修改时需要保持以下基本消息类型稳定：

```json
{"type":"register","node_id":"...","client_token":"...","platform":"...","tools":[]}
{"type":"register_ack","ok":true}
{"type":"heartbeat","node_id":"..."}
{"type":"heartbeat_ack","ts":1710000000.0}
{"type":"invoke","id":"req-...","tool":"...","arguments":{}}
{"type":"invoke_result","id":"req-...","success":true,"result":{}}
```

修改注册逻辑时，必须保留工具名冲突检查：

- server 本地工具名优先。
- 不同 client 之间不能重名。
- 同一个 client 重新注册时，可以替换自己的旧工具映射。

相关测试在 `tests/test_cluster.py`。

## 修改组件目录逻辑

`catalog_core.py` 负责远程组件目录。关键规则：

- `repo_url` 必须能解析为 GitHub owner/repo。
- 优先读取 `index.json`。
- 读取失败时退回 GitHub contents API。
- 组件名使用 `safe_component_name()` 清洗，只允许字母、数字、下划线和短横线。
- 安装时只写入 `component.py`、`README.md` 和缺失的 `__init__.py`。

测试方向：

- 非法 repo URL。
- 非法组件名。
- `index.json` 缺失时的 fallback。
- README 平台行解析。
- 安装路径是否正确。

## 开发组件

新组件通常应放到 `xiaozhiMCP-components` 仓库。最小结构：

```text
my_component/
  __init__.py
  component.py
  README.md
```

最小代码：

```python
class Component:
    def __init__(self, config=None, full_config=None):
        self.config = config or {}

    def supports_role(self, role: str) -> bool:
        return role == "client"

    def register(self, mcp) -> None:
        @mcp.tool()
        def my_component_ping() -> dict:
            return {"success": True, "message": "pong"}

    def export_tools(self) -> list[dict]:
        return [
            {
                "name": "my_component_ping",
                "description": "Return pong from this component.",
                "input_schema": {"type": "object", "properties": {}},
            }
        ]

    def invoke_tool(self, tool_name: str, arguments: dict) -> dict:
        if tool_name != "my_component_ping":
            raise RuntimeError(f"unsupported tool: {tool_name}")
        return {"success": True, "message": "pong"}
```

README 最后一行必须声明平台：

```text
Platforms: Windows|Linux|MacOs
```

并在组件仓库 `index.json` 添加条目。

## 测试建议

主仓库已有测试覆盖以下方向：

- 集群工具名冲突。
- 同一 client 重新注册替换旧工具映射。
- 目录核心逻辑。
- 旧版 MCP router 消息格式兼容。

新增功能建议至少补充：

- 正常路径测试。
- 配置缺失或非法输入测试。
- 网络或远程调用失败时的错误返回测试。
- 与角色过滤相关的测试。

运行：

```bash
uv run pytest
```

只运行某个测试文件：

```bash
uv run pytest tests/test_cluster.py
```

## 调试技巧

启用 server 日志后，关键日志会输出到标准错误或控制台。常见调试点：

- CLI 是否找到预期配置文件。
- 自动识别出的 role 是否正确。
- 组件发现数量和导出工具数量。
- client 是否收到 `register_ack`。
- `cluster_list_clients()` 是否看到目标节点。
- `logmcp_get_errors()` 是否记录注册错误。

调试远程调用时，可以先列工具再调用：

```python
cluster_list_remote_tools()
cluster_call_remote_tool(tool_name="clipboard_get", arguments_json="{}")
```

## 发布前检查

提交前建议执行：

```bash
uv run pytest
uv run ruff check .
```

如果改动了配置字段，还应同步更新：

- `config.example.toml`
- `config_guide.py`
- README 或 Wiki 配置说明

如果改动了组件接口，还应同步更新：

- `runtime.py` 中的加载逻辑
- 组件仓库 README
- 组件开发示例
- 相关测试
